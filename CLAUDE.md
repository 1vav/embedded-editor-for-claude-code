# embedded-editor-for-claude-code — Developer Guide

This file is for contributors working on the package itself, not for users of the package.
(Users get their own `CLAUDE.md` generated by `npx embedded-editor-for-claude-code init`.)

## Git / PR Workflow Rules

### Rule 1 — Never push to a merged/closed branch

Before ANY push, verify the branch's PR is still open:
```sh
gh pr list --repo 1vav/embedded-editor-for-claude-code --state all | grep <branch-name>
```
If status is MERGED or CLOSED → **stop**. Create a new branch from `main`:
```sh
git checkout -b fix/descriptive-name origin/main
```
GitHub squash-merges orphan any new commits on the old branch. They never reach main.

**Rule: one branch = one PR = done. Never reuse after merge.**

### Rule 2 — Always rebase, never stash-pop onto a new branch

When switching to a fresh branch to carry forward a change:
1. `git checkout -b fix/new-name origin/main`
2. Make the edit directly (Read → Edit/Write tools)
3. Commit and push

**Never** `git stash && checkout new-branch && git stash pop`. Stash-pops create merge conflicts when the stash was built against a different base. Write the change fresh instead.

### Rule 3 — Keep PRs to a single focused file change

When the same file is being iteratively fixed across multiple PRs, conflicts accumulate. If a fix attempt fails, understand the root cause completely before opening the next PR — don't make incremental guesses.

## Architecture

Three components run together:

| Component | File | Transport |
|-----------|------|-----------|
| MCP server | `src/server.js` | stdio JSON-RPC (launched by Claude Code) |
| HTTP viewer server | `src/viewer-server.js` | HTTP + SSE on port 3000 |
| Browser SPA | `src/viewer/entry.jsx` → `vendor/viewer.js` | served by the HTTP server |

The MCP server and HTTP server are started independently. Claude Code launches the MCP server via stdio; the user starts the HTTP server via `/editor-start` or `node bin/cli.js view`.

## PNG Rendering Pipeline

`src/render.js` renders Excalidraw diagrams to PNG:

1. Load `vendor/excalidraw.mjs` (pre-bundled — avoids ESM issues in Node)
2. Call `exportToSvg(elements, appState, files)` inside a `jsdom` shim
3. Rasterise SVG → PNG with `@resvg/resvg-js`
4. Return as base64 string

The render runs in the same Node process. A 15-second `Promise.race` timeout in `previewContent()` prevents a stuck render from hanging the MCP server.

## Shared Utilities (`src/workspace.js`)

Both `server.js` and `viewer-server.js` import from `src/workspace.js`:

- `ROOT` — resolved once from `EXCALIDRAW_ROOT` env or `cwd`
- `validateName(raw, stripExt)` — strict path sanitiser; rejects traversal and special chars
- `resolveFile(name, ext)` — resolves a validated name to an absolute path inside ROOT
- `rewriteLinks(oldName, newName)` — rewrites all `[[wikilinks]]` in `.md` and `.excalidraw` files
- `findBacklinks(name)` — finds all files that link to a given name
- `listSnapshots(name)` — lists saved history snapshots for a diagram

Any security fix to path validation goes in `workspace.js` and propagates everywhere automatically.

## Security Notes

- **CSRF:** all `PUT`/`POST`/`DELETE` requests require an `Origin` header matching localhost. Absent origin → 403.
- **Path traversal:** `validateName` rejects `..`, `.`, absolute paths, and non-`[\w\-. ]` segments.
- **XSS:** markdown HTML is sanitised with DOMPurify before being set via `dangerouslySetInnerHTML`.
- **Body limit:** 5 MB max request body. 30-second request timeout.
- **SSE CORS:** `Access-Control-Allow-Origin` is set to the request origin (not `*`).

## Adding a New File Type — Icon Sync Rule

When adding a new file type with a brand icon (e.g., `DuckBrandIcon`), it must appear in **both** of these locations in `src/viewer/entry.jsx`:

1. **BrandMark component** (~line 938) — the title-bar icon strip shown at the top of every page.
2. **Home page icon row** (~line 1948) — the large icon display shown when no file is open.

Also update the **home page subtitle** (~line 1958) to include the new type (e.g., add `· tables` to the `diagrams · canvases · notes · ...` string).

Forgetting either location results in the icon appearing in one place but not the other.

## Starting the Dev Server

**Always use `preview_start` — never `node bin/cli.js view` directly.** Running the CLI directly occupies port 3000 and causes `preview_start` to fail with `EADDRINUSE`.

Reliable restart sequence:

1. Kill anything already on port 3000:
   ```sh
   lsof -ti:3000 | xargs kill -9 2>/dev/null; sleep 1
   ```
2. Call `preview_start` with name `"Embedded Editor"` (defined in `.claude/launch.json`).

If `preview_start` reports the server is already running (`reused: true`), it's healthy — skip step 1.

## Dev Workflow: Making SPA Edits Visible

Two gotchas that will silently hide your `src/viewer/entry.jsx` changes:

**1. `.claude/launch.json` must point to the local `bin/cli.js`, not the global install.**

The `editor-start` skill writes the global path by default (so end-users get `/editor-start` working immediately). When developing the package, fix it:

```json
"runtimeArgs": ["/Users/vaibha/Downloads/git/embedded_editor_for_claude_code/bin/cli.js", "view"]
```

If the wrong path is set, edits to `src/viewer/entry.jsx` are ignored — the globally installed `vendor/viewer.js` is served instead. Verify by curling the live server and grepping for a newly added CSS class name.

**2. The browser caches `vendor/viewer.js` for one year (`Cache-Control: immutable`).**

Cache busting is tied to `PACKAGE_VERSION` (the `?v=X.Y.Z` query string on the script tag). If the version doesn't change between rebuilds, the browser serves the cached bundle even after a server restart.

Fix: temporarily add a `-dev` suffix to the version in `package.json` before rebuilding:

```sh
# bump to bust cache
node -e "const fs=require('fs'),p=JSON.parse(fs.readFileSync('package.json','utf8'));p.version=p.version.replace(/-dev$/,'')+'-dev';fs.writeFileSync('package.json',JSON.stringify(p,null,2)+'\n');console.log(p.version)"

npm run build:viewer
lsof -ti:3000 | xargs kill -9 2>/dev/null; sleep 1
# then preview_start
```

Verify the new bundle is live:
```sh
curl -s --compressed http://127.0.0.1:3000/vendor/viewer.js | grep -c "your-new-class-name"
```

Revert the version suffix before releasing.

## npm Publish — OIDC Trusted Publisher

Releases are fully automated via `.github/workflows/release-on-merge.yml`. **Never run `npm publish` locally.**

The pipeline uses npmjs.com Trusted Publisher (OIDC) — no `NPM_TOKEN` secret is stored anywhere.

### How it works

1. The job declares `environment: release`, which makes the GitHub JWT `sub` claim include `environment:release` — required to match the trusted publisher config on npmjs.com.
2. `actions/github-script` requests a JWT with audience **`npm:registry.npmjs.org`** (not `npm:publish` — that's wrong and silently breaks everything).
3. The JWT is exchanged at `POST https://registry.npmjs.org/-/npm/v1/oidc/token/exchange/package/embedded-editor-for-claude-code` (empty body, JWT as `Authorization: Bearer`) → returns a short-lived npm token.
4. That token is written to `NODE_AUTH_TOKEN`, overwriting what `setup-node` injected.
5. `npm publish --access public --provenance` runs normally.

### Hard-won lessons — do not repeat

| Mistake | Symptom | Why wrong |
|---------|---------|-----------|
| Audience `npm:publish` | `ENEEDAUTH` or `E404` from registry | Registry requires `npm:registry.npmjs.org` |
| Passing the raw JWT as `NODE_AUTH_TOKEN` | `E404: not in this registry` or HTTP 401 | Registry doesn't accept OIDC JWT as bearer; must exchange first |
| Missing `environment: release` on the job | Token exchange 403 | JWT `sub` won't match trusted publisher config |
| `npm install -g npm@latest` in CI | `MODULE_NOT_FOUND: promise-retry` | Self-upgrade breaks npm; never upgrade npm in CI |
| Deleting `.npmrc` | `ENEEDAUTH` immediately | `.npmrc` provides the `_authToken` wiring; removing it breaks auth entirely |

### To trigger a release

1. Bump `version` in `package.json` (no `-` suffix — pre-release versions are skipped).
2. Merge to `main`. The workflow fires automatically.

## Performance Guidelines

These patterns are already in the codebase — preserve them when editing.

### Server (`src/viewer-server.js`)

- **Vendor assets are pre-loaded at startup** into an in-memory Map (`vendorFiles`). Do not add `fs.readFile` calls inside the vendor request handler — populate the Map in `startViewerServer` instead.
- **Pre-compressed variants** (`.gz`/`.br`) are generated by the build script and served with `Content-Encoding`. Respect the `Accept-Encoding` check in the vendor handler.
- **Per-diagram render locks** (`renderLocks` Map, not a global lock). New render paths must call `queueRender(name, fn)` with the diagram name.
- **PNG cache stores `Buffer`** directly — `renderToPng()` (returns Buffer) not `renderToPngBase64()`. Never store base64 in `pngCache`.
- **LRU cap** — call `trimCache(cache)` after every `set()` on `svgCache` or `pngCache`.
- **ETags on SVG/PNG** — always emit `ETag` + `Last-Modified` and honour `If-None-Match` for 304 responses.
- **SSE broadcasts carry `op`** — always pass `op: "created" | "updated" | "deleted" | "renamed"` so clients pick the cheapest refresh.

### Workspace (`src/workspace.js`)

- `findBacklinks` and `rewriteLinks` fan-out file reads/writes via `Promise.all`. Keep it that way — don't introduce sequential `await` in loops.

### SPA (`src/viewer/entry.jsx`)

- `refreshRecent()` fetches only `/api/recent` (no glob scans). Use it for saves of *existing* files where the file lists don't change. Use the full `refresh()` only for creates, deletes, and renames.
- SSE handler routes `op === "updated"` → `refreshRecent()`, everything else → `refresh()`. Keep this distinction.
- `handleUserSave` calls `refreshRecent()`, not `refresh()`.

### Build (`scripts/build-viewer-bundle.mjs`)

- Post-build compression (gzip + brotli) is part of `npm run build:viewer`. If you add new large assets, compress them too.
- `vendor/` is gitignored — compressed variants land there safely (no API key risk). They are distributed via npm `files[]`.
